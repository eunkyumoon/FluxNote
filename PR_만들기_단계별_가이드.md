# GitHub PR 만들기 - 화면별 단계별 가이드

현재 GitHub의 Pull requests 페이지에서 PR을 만드는 방법을 단계별로 설명합니다.

---

## 🎯 현재 화면에서 시작하기

지금 보이는 화면:
- "Pull requests" 탭이 선택되어 있음
- "New pull request" 버튼이 보임
- "Welcome to pull requests!" 메시지가 표시됨

---

## 📝 단계별 가이드

### 1단계: "New pull request" 버튼 클릭

**위치**: 화면 오른쪽 상단의 **초록색 "New pull request"** 버튼

1. 마우스로 **"New pull request"** 버튼을 클릭합니다
2. 새로운 페이지로 이동합니다

---

### 2단계: 브랜치 선택 화면

새 페이지에서 보이는 것:
- **"base:"** 드롭다운 (왼쪽)
- **"compare:"** 드롭다운 (오른쪽)
- 두 브랜치 간의 차이점이 표시됨

**해야 할 일**:

1. **"base:"** 드롭다운 클릭
   - `main` 선택 (병합할 대상 브랜치)
   - 이미 `main`이 선택되어 있을 수 있음

2. **"compare:"** 드롭다운 클릭
   - `dev` 선택 (병합할 소스 브랜치)
   - 드롭다운에서 `dev` 브랜치를 찾아 클릭

3. 브랜치를 선택하면:
   - 두 브랜치 간의 차이점이 자동으로 표시됩니다
   - 변경된 파일 목록이 보입니다
   - "Able to merge" 또는 "Can't automatically merge" 메시지가 표시됩니다

---

### 3단계: PR 정보 입력

브랜치를 선택하면 아래로 스크롤하면 입력 필드가 보입니다:

#### 3-1. 제목 (Title) 입력

**위치**: 가장 위쪽의 큰 입력창

**예시 제목**:
```
dev 브랜치의 변경사항을 main에 병합
```

또는

```
새로운 기능 추가 및 개선사항
```

**팁**: 
- 명확하고 간결하게 작성
- 무엇을 변경했는지 한눈에 알 수 있게

#### 3-2. 설명 (Description) 입력

**위치**: 제목 아래의 큰 텍스트 영역

**예시 설명**:
```markdown
## 변경사항
- dev 브랜치에서 작업한 내용을 main에 병합합니다

## 주요 내용
- 프로젝트 초기 설정
- 메모 앱 기능 구현

## 테스트
- 로컬에서 테스트 완료
```

**팁**:
- 변경 이유와 내용을 설명
- 마크다운 문법 사용 가능 (## 제목, - 리스트 등)

---

### 4단계: 추가 옵션 설정 (선택사항)

PR 정보 입력 필드 아래에 여러 옵션이 있습니다:

#### 4-1. Reviewers (리뷰어)
- 다른 사람이 코드를 리뷰하도록 지정
- 혼자 작업하는 경우 생략 가능

#### 4-2. Assignees (담당자)
- PR을 담당할 사람 지정
- 자신을 지정하거나 생략 가능

#### 4-3. Labels (라벨)
- PR에 라벨 추가
- 예: `enhancement` (기능 개선), `bug fix` (버그 수정)
- 선택사항

#### 4-4. Projects (프로젝트)
- 프로젝트 보드에 연결
- 선택사항

---

### 5단계: PR 생성

1. 모든 정보를 입력한 후
2. 페이지 하단으로 스크롤
3. **"Create pull request"** 버튼 클릭
   - 버튼은 초록색 또는 파란색일 수 있습니다
4. ✅ PR이 생성되었습니다!

---

### 6단계: PR 확인

PR이 생성되면:
- PR 페이지로 자동 이동합니다
- PR 번호가 표시됩니다 (예: #1)
- PR 상태가 "Open"으로 표시됩니다

---

### 7단계: PR 병합 (Merge)

PR을 병합하려면:

1. PR 페이지에서 아래로 스크롤
2. **"Merge pull request"** 버튼 클릭
   - 초록색 버튼입니다
3. 병합 메시지 확인 (또는 수정)
4. **"Confirm merge"** 버튼 클릭
5. ✅ 병합 완료!

**병합 후**:
- "Pull request successfully merged" 메시지가 표시됩니다
- PR이 "Merged" 상태로 변경됩니다
- `main` 브랜치에 `dev` 브랜치의 변경사항이 반영됩니다

---

## 🎨 화면별 상세 설명

### 브랜치 선택 화면에서 보이는 것들

```
┌─────────────────────────────────────────┐
│  base: main          compare: dev       │
│  [드롭다운]          [드롭다운]          │
└─────────────────────────────────────────┘
         ↓
    변경사항 표시
    - 파일 목록
    - 추가/삭제된 줄 수
    - "Able to merge" 상태
```

### PR 정보 입력 화면

```
┌─────────────────────────────────────────┐
│  제목 입력창                              │
│  [dev 브랜치의 변경사항을 main에 병합]    │
└─────────────────────────────────────────┘
         ↓
┌─────────────────────────────────────────┐
│  설명 입력창                              │
│  [변경사항 설명...]                       │
└─────────────────────────────────────────┘
         ↓
    [Reviewers] [Assignees] [Labels] ...
         ↓
    [Create pull request] 버튼
```

## ⚠️ 주의사항

### 1. "Can't automatically merge" 메시지가 나올 때

**의미**: 자동으로 병합할 수 없음 (충돌 발생)

**해결 방법**:
1. 로컬에서 충돌 해결
2. 다시 푸시
3. PR이 자동으로 업데이트됨

### 2. 변경사항이 보이지 않을 때

**원인**: `dev` 브랜치에 변경사항이 없거나 이미 `main`에 병합됨

**해결 방법**:
1. `dev` 브랜치에 변경사항이 있는지 확인
2. 변경사항을 푸시했는지 확인

### 3. "Create pull request" 버튼이 비활성화될 때

**원인**: 제목이 비어있거나 필수 정보가 누락됨

**해결 방법**:
1. 제목 입력 확인
2. 필수 필드 모두 입력 확인


## 💡 실전 팁

### 팁 1: PR 전에 미리보기
- 브랜치를 선택하면 변경사항을 미리 볼 수 있습니다
- 파일별로 변경 내용을 확인할 수 있습니다

### 팁 2: 작은 단위로 PR 만들기
- 한 번에 너무 많은 변경사항을 포함하지 않기
- 하나의 기능이나 버그 수정 단위로 PR 만들기

### 팁 3: 명확한 제목
- PR 제목만 봐도 무엇을 변경했는지 알 수 있게
- 예: "버그 수정: 검색 기능 오류 해결"

### 팁 4: 스크린샷 첨부 (UI 변경 시)
- UI가 변경된 경우 설명에 스크린샷 첨부
- 이미지를 드래그 앤 드롭으로 추가 가능
